# Learn Git Branching을 통해서 git 배우기

## 개요

이번 단계에서는 지금까지 학습한 Git 개념을 기반으로  
원격 저장소(Remote Repository) 와의 협업 흐름을 학습했다.  
이전 단계까지는 로컬에서 브랜치와 커밋을 다루는 방법에 집중했다면,  

이번 단계에서는  

- 다른 사람이 작업한 내용을 가져오고  
- 내 작업을 공유하고  
- 충돌을 해결하는 과정  

즉, 실제 협업 상황에서의 Git 사용 방식이 핵심이었다.  
특히 단순 명령어 사용이 아니라  
“**언제 fetch / pull / push를 써야 하는지” 판단하는 능력**이 중요해지는 단계였다.

## 원인

Git은 히스토리가 꼬이는 것을 방지하기 위해 **원격보다 뒤쳐진 상태에서는 push를 막는다.**

## 해결 방법

### 방법 1

```python
git fetch
git rebase o/main
git push
```
  
내 커밋을 최신 위로 재정렬  

---

### 방법 2

```python
git fetch
git merge o/main
git push
```
  
merge commit 생성  

---

### 방법 3  

```python
git pull --rebase
git push
```
  
실무에서 가장 많이 사용

## 비교

### fetch vs pull  

명령어  / 설명  
  
fetch	  / 데이터만 가져온다.  
pull	  / 가져오고 병합한다.  

---

### pull vs pull --rebase  

방식	  / 특징  
  
pull	  / merge 방식  
pull --rebase	  / 히스토리 정리  

---

### 내용 정리  

#### 공식

```python
git pull --rebase
git push
```
  
Git 협업 핵심 흐름  

 ```python 
fetch → rebase → push
```

## 이번 단계에서 느낀 점  
  
이번 단계는 단순히 Git 명령어를 사용하는 것을 넘어서  
협업 상황에서 Git이 어떻게 동작하는지 이해하는 단계였다.   
  
특히 다음과 같은 점이 인상적이었다.  
  
- push가 단순 업로드가 아니라 “조건이 있는 동작”이다.  
- fetch와 pull의 차이를 명확하게 이해하게 된다.  
- rebase를 사용하면 히스토리를 더 깔끔하게 유지할 수 있다.  
- 협업에서는 항상 원격 상태를 기준으로 작업해야 한다.  
  
처음에는 왜 push가 막히는지 이해가 되지 않았지만,  
히스토리 흐름을 따라가며 생각하니 Git의 구조가 점점 명확해졌다.  

---

## 정리
  
이번 단계에서 학습한 내용은은 다음과 같다.  
  
- 원격 저장소와 로컬 저장소는 별개이다.
- fetch는 안전하게 데이터를 가져오는 명령어이다.
- pull은 fetch + merge이다.
- pull --rebase는 히스토리를 깔끔하게 유지한다.
- push는 원격보다 뒤쳐져 있으면 실패한다.
- 협업 시 rebase 기반 작업이 중요하다.
